## 线上故障：新闻概览停在 7/16 中午，23:26 才发现（已修复）
- **现象**：新闻 `published_at` 最新停在 `2026-07-16 04:27 UTC`(=北京12:27)，推荐流/概览卡在中午，晚间无新料。
- **根因**：当天中午 docker 迁移时 `systemctl stop docker` 对 alpha-web 发了 **SIGTERM 强杀**。调度器用 Redis 分布式锁 `alphareader:scheduler_lock`(TTL=3600s) 保证单 worker 跑管线（见 `backend/app/services/scheduler.py`）。旧 worker 被强杀时后台释放锁的线程没跑完→留下 **stale 锁**；而 worker 只在**启动时** `set nx` 尝试拿锁一次、**永不重试**。新容器 12:44 起来后两 worker 都判"别人持有"→跳过 `start_scheduler`→调度器永久停摆。锁 1h 后虽过期，但无人重试，故直到 23:26 仍无新闻。
- **验证锁状态**：`docker exec alpha-cache redis-cli -a $REDIS_PASSWORD GET/TTL alphareader:scheduler_lock` → TTL=-2（键已不存在，确认是"启动期一次性拿锁失败+不重试"的设计缺陷，而非锁本身持续阻塞）。
- **修复**：`docker restart alpha-web`（重启后新 worker 启动时锁已空闲→`set nx` 成功→拉起调度器）。日志见 `本 worker 已获取调度锁，启动定时任务调度器` + `Pipeline job triggered`。
- **结果**：重启后管线立即触发，跑 `fetched=214, scored=75, stored=75`（耗时 244s，因 LLM 限流+英文两段式翻译重试偏慢，属正常），DB `max(published_at)` 推进到 23:28 北京，新闻恢复每 15min 滚动。
- **运维经验**：
  - **任何 `docker restart alpha-web` / 容器重启后，若发现新闻停更，先查 web 日志是否出现"未获取调度锁（其他 worker 已持有）"——若出现即调度器未起，再 restart 一次即可（锁 TTL 1h，重启后通常已空闲）。**
  - **设计隐患（待修）**：worker 仅在启动时尝试拿锁、不重试。若重启时恰有 stale 锁（如 SIGTERM 未释放），调度器会静默停摆且无告警。建议：①shutdown 时更可靠释放锁（如 atexit/信号钩子同步删 key）；②worker 加一个周期性"若调度器未运行则重试拿锁"的后备任务；③调度器未启动时发 alert。优先级中，先靠"重启 web"兜底。
- **已修复（2026-07-16 23:47 部署，commit `3bdfbc8` 已 push + 老服务器重建 web）**：`backend/app/services/scheduler.py` 加调度器**自愈**：①锁值写入本 worker 唯一 owner id(`{pid}-{uuid4[:8]}`)，`_WORKER_ID`；②未持锁 worker 启动 `_lock_watchdog_loop`，每 `_WATCHDOG_INTERVAL=60s` 做 `set nx` 重试，锁空闲即接管并调 `_run_scheduler_jobs()`（原 `start_scheduler` 主体抽成此函数）；③持锁 worker 启动 `_lock_keepalive_loop` 每 `TTL//3` 续期，发现锁丢失则 `_stop_local_scheduler()` 让出；④`_release_scheduler_lock_bg` 改为仅删「自己持有」的锁（防多 worker 互相误删）。
- **验证**：web 重建后日志 — worker1 "本 worker 已获取调度锁，启动定时任务调度器" + `Scheduler started` + pipeline 立即触发；worker2 "本 worker 未获取调度锁…启动自愈看门狗"；`redis-cli GET alphareader:scheduler_lock` = `8-e5eecf1a`（owner 格式）。**今后再遇 stale 锁致调度器停摆，最坏 `锁剩余TTL + 60s` 内自动恢复，无需人工 restart。** 运维经验第 53 行「restart 兜底」可作废，但人工 restart 仍可用。
