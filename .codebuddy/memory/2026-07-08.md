  - 实测 `md5(md5_hex)`、`md5(md5_bytes)` 均报 10012；`md5(sha1_hex)` 返回 errno:0。
- 修复（commit 714e5fe）：`rss_fetcher.py` 新增 `_cls_sign` / `_cls_build_params`，`FeedSource` 加 `signed_cls` 标志，`_fetch_raw_items` 增加签名分支；财联社 `url` 改为 `v1/roll/get_roll_list` 并置 `signed_cls=True`。`_parse_cls` 已兼容该结构，无需改。
- ⚠️ 该接口已第三次改版（updateTelegraphList → telegraphList → v1/roll/get_roll_list + 签名），未来若再失效，直接逆向 cls.cn 前端 JS 的 `_cls_sign` 与 `sv` 版本号即可，无需再全网试错。

## 前端首页全空白根因与修复（23:0x）

用户报首页新闻全空，浏览器 Console 报 `request:fail timeout`（uni.request 15s 超时，Nginx 记 499）。

**根因**：backend `gunicorn --workers 1`。pipeline 每 15 分钟跑批（抓 24 源 + DeepSeek 打分 + 向量去重），CPU 密集任务阻塞唯一 event loop，期间浏览器 API 请求排队 >15s 超时 → 前端 `newsList` 空 → 页面空白。本地 curl 测 API 仅 0.05s，但浏览器经公网超时。

**修复（commit 1d70954）**：
- `backend/Dockerfile`：`--workers 1` → `--workers 2`（服务器 4 核 / 3.3GB，web mem_limit 1536m，2 worker 实测 ~600-700MB 安全）。
- `backend/app/services/scheduler.py`：新增 Redis 分布式锁（`alphareader:scheduler_lock`，SET NX + TTL 3600s）。`start_scheduler()` 改为 `async`，仅成功获取锁的 worker 启动调度器，其余 worker 专责 API，避免多 worker 重复跑 pipeline。`stop_scheduler()` 后台线程释放锁。Redis 不可用时 fail-open（仍启动，去重层兜重复）。
- `backend/app/main.py`：lifespan 内 `await start_scheduler()`（原先同步调用会因事件循环已运行而报 "event loop is already running"）。
- `frontend/src/utils/api.js`：请求 `timeout` 15s → 30s（兜底）。

**验证**：部署后日志显示一 worker "未获取调度锁跳过"，另一 worker "已获取调度锁启动调度器" + `Pipeline run starting`（pipeline 只跑一次）；pipeline 跑批期间连续 5 次 API 请求仍 0.03~0.12s 返回 200。

**关键经验**：
- gunicorn 单 worker + 重后台任务会饿死 API；多 worker 是标准解，但 APScheduler 需配分布式锁防重复。
- 排查 API 超时优先看 Nginx `access.log` 的 499（客户端断开）vs 200，以及 backend `GET /api/v1/health` 的耗时（跑批时会飙到 14s，证明 event loop 被阻塞）。
- 浏览器 Console 的 `request:fail timeout` + 服务端 API 本地快 = 链路中间（worker 阻塞 / 限流 / HTTP/2）问题，不是数据或前端渲染问题。

⚠️ **重要更正（23:5x）**：上面把「首页空白」根因归为 worker 超时，但 worker 修复（commit 1d70954）只是消除了超时；**页面在后端恢复正常后仍空白的真因是前端 `onShow` 注册方式错误**，与超时无关：
- 首页用 `instance.proxy.$options.onShow = [...]` 手动注册页面生命周期，但 uni-app H5 运行时**只派发通过官方 `onShow()` 注册的实例钩子**，从不读取 `$options.onShow`。
- 实证：运行时 bundle 中 `$options.onShow` 出现 0 次；派发处代码为 `const{onShow:n}=e; n&&R(n)`（读实例上的 `onShow`，非 `$options.onShow`）。
- 现象：无 console 报错、Network 里看不到页面的新闻请求（用户贴的 JSON 是手动测接口，非页面请求），页面恒空。
- **修复（commit 45ef08c）**：`frontend/src/pages/index/index.vue` 改用官方 `onShow/onHide/onPullDownRefresh/onReachBottom`（从 `@dcloudio/uni-app` 导入）替换 `$options.*` hack。全仓仅首页有此 hack，其余页面用官方钩子正常。
- 排查方法：怀疑页面不加载时，先确认生命周期钩子是否真的被框架派发（grep 运行时 bundle 是否读取 `$options.onX`），而不是只盯着数据/渲染。已部署验证：前端容器重建后首页恢复正常。
